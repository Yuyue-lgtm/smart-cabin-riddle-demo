# 主持话术归属表

这份文档专门回答一个问题：

当前页面里，AI 主持人的话术到底有多少是 Coze Workflow 生成的，又有多少是前端为了稳定演示而覆盖或改写的。

目标不是立刻删掉所有前端文案，而是先把边界划清楚，方便后面做两件事：

1. 降低重复度
2. 把真正应该由 Coze 决定的主持风格，逐步还给 Coze

## 当前结论

当前主持话术的来源分成三层：

| 层级 | 来源 | 当前职责 |
| --- | --- | --- |
| 第一层 | Coze Workflow `ai_reply_text` | 正常游戏中的主要主持话术来源 |
| 第二层 | 前端改写 Coze 文案 | 删除出戏表达、保护真实副驾主流程 |
| 第三层 | 前端本地覆盖 | 安全类事件、答对兜底、超时/失败 fallback、黄金线本地 cue |

所以现在“重复度偏高”并不是单一原因，而是这三层叠加：

- Coze 当前策略还不够多，普通问答容易复用同一句式
- 前端为了稳定演示，在几个关键节点直接接管了主持话术
- 某些 fallback 文案是写死的，自然会反复出现

## A 类：必须保留的稳定性覆盖

这些内容现在不建议直接放权给 Coze。它们承担的是“演示不翻车”的责任，而不只是文案责任。

| 位置 | 触发场景 | 当前前端行为 | 为什么必须保留 |
| --- | --- | --- | --- |
| `applyImmediateSafetyPause()` | `hard_brake` | 立即说“大家坐稳，游戏先暂停。” | 安全事件必须本地即时接管，不能等 Workflow |
| `applyImmediatePassengerSleep()` | `passenger_sleep` | 立即切轻声模式，不 cue 睡着乘客 | 睡着乘客不能被 Workflow 误 cue |
| `applyImmediateDriverTired()` | `driver_tired` | 立即把答题权转给副驾和后排 | 主驾疲惫属于安全相关降级 |
| `applyImmediateNearDestination()` | `near_destination` | 立即进入收尾局状态 | 节奏收束需要稳定，不适合等模型自由发挥 |
| `localDecision()` | Workflow 超时/失败/返回异常 | 使用本地 fallback 文案 | 这是页面不白屏、不卡死的最后保险 |
| `applyAnswerHitGuard()` | 用户输入明确包含谜底 | 前端直接判定为答对 | 避免 Workflow 偶发误判导致“明明答对却没过” |
| `normalizeHostReplyForRealUser()` | Workflow 想 cue 左后/右后，但真实用户固定副驾 | 改写为“副驾继续接上” | 这是当前体验模式的结构保护，不只是文案偏好 |
| `sanitizeHostReplyText()` 中的“请选择主题”清洗 | Workflow 仍输出不存在的主题选择文案 | 前端删掉出戏句子 | 在 Coze 修好之前，这层保护必须保留 |

### 这类内容的建议

- 保留状态接管
- 文案可以更短、更中性
- 后续如果 Coze 稳定，再逐步减少“完整句子覆盖”，保留“状态保护”即可

## B 类：建议继续由前端持有，但可以缩短文案

这些话术不一定要长期写死在前端，但它们目前更多是在承担“流程提示”的作用，而不是“人格表达”。

| 位置 | 场景 | 当前状态 | 建议 |
| --- | --- | --- | --- |
| `cueRealUser()` | 黄金体验线里主动 cue 副驾 | 由前端写死 | 短期保留，因为它是时间轴脚本的一部分 |
| `sendQuestion()` 忙时提示 | 上一轮未返回，用户继续输入 | “这条问题我先记下...” | 可以继续前端持有，但改成更短状态提示 |
| `processNextPendingChat()` | 排队问题开始处理 | “刚才那条问题接上了...” | 这句已经弱化为 thinking UI，可继续保持无文案 |
| `getPrestartHostText()` | 开始模拟前的待开局状态 | 当前是前端预置开场邀请 | 可以保留为体验模式固定开局，不必强求 Coze 接管 |

### 这类内容的建议

- 不需要完全删掉
- 但应该尽量从“完整主持台词”收缩为“状态提示”或“脚本 cue”
- 这样就不会和 Coze 的人格化表达打架

## C 类：可以放权给 Coze 的文案覆盖

这些内容本质上是“主持风格”，不是“状态保护”。如果想降低重复度，优先应该从这里释放给 Coze。

| 位置 | 场景 | 当前前端状态 | 建议归属 |
| --- | --- | --- | --- |
| `makeVictoryHostText()` | 用户答对后的总结 | 当前由前端统一生成 | 建议下一步放给 Coze，只保留胜利状态保护 |
| `localDecision()` 里的普通提问 fallback | Workflow 异常时普通 yes/no 提示 | 当前是固定句式 | 可以保留 fallback，但增加 4-6 条本地轮换模板 |
| `localDecision()` 里的 `start_game` 文案 | 开场介绍 | 当前 fallback 固定 | 长期建议由 Coze 负责正式主持风格 |
| `localDecision()` 里的 `resume_game` 文案 | 恢复游戏 | 当前 fallback 固定 | 可让 Coze 决定恢复话术，前端只保留恢复状态 |
| `localDecision()` 里的 `environment_change` 文案 | 环境切换说明 | 当前固定句式 | 建议交给 Coze，根据环境自由改写 |
| `localDecision()` 里的 `speed_change` 文案 | 高速安全提醒 | 当前固定句式 | 结构要保留，语气可交给 Coze |

## 前端当前真正会“压过”Coze 的地方

如果只看“最终显示给用户的主持话术”，现在真正会覆盖 Coze 的点主要是：

| 覆盖函数 | 会不会压过 Coze 原文 | 说明 |
| --- | --- | --- |
| `sanitizeHostReplyText()` | 会 | 删除“请选择主题”之类的出戏句子 |
| `normalizeHostReplyForRealUser()` | 会 | 把“请左后提问”改成“副驾继续接上” |
| `applyAnswerHitGuard()` | 会 | 用户显式答对时直接进入胜利输出 |
| `makeVictoryHostText()` | 会 | 胜利总结完全由前端生成 |
| `localDecision()` | 会 | 只在 Workflow 失败/超时时接管 |

## 优先处理建议

如果目标是“先快速降低重复度”，建议按这个顺序做：

1. 先减少前端完整句子覆盖
   当前最值得先放权的是 `makeVictoryHostText()`。

2. 再增强 Coze 提示词里的“表达轮换”
   特别是 `S00`、`S02`、`S14`、`S21` 这几类高频策略。

3. 最后再收缩 fallback 文案
   fallback 仍然要保留，但可以做成本地多模板轮换，而不是单句写死。

## 建议的下一步边界

现阶段比较稳妥的分工建议是：

### 前端保留

- 安全接管
- 睡着/疲惫/目的地收尾的状态保护
- 真实副驾主流程保护
- 超时和错误 fallback
- 气泡显示时序保护

### Coze 接管

- 正常问答主持语气
- 环境融合表达
- 卡题提示表达
- 接近答案表达
- 答对后的高情绪价值总结
- 不同乘客关系下的口吻变化

## 一个实用判断标准

后面每遇到一条“前端想写主持文案”的需求，可以直接用这个标准判断：

- 如果这句话是在保护流程不乱，前端保留
- 如果这句话是在体现主持人格，优先交给 Coze
