# 产品简报

## 一句话定位

智能座舱 AI 猜谜 Demo 是一个可分享体验的车内 AI 游戏原型。系统通过时间轴模拟车外环境、车辆状态和乘客状态变化，由 Workflow 驱动 AI 主持人与半自主模拟乘客参与游戏，用户可以在车内互动区参与提问，网页实时表现座舱、游戏和事件打断效果。

## 产品目标

本项目同时服务两个场景：

- 内部 AI 应用比赛现场展示
- 部门同事通过公网链接自主体验

因此它不能只是一条固定演示链路，也不能完全依赖演示者手动模拟全车乘客。目标是让用户打开链接后，即使只有自己一个真实用户，也能感受到“车里有其他乘客一起玩，AI 主持人在控场”。

## 核心价值

- AI 不只是回答问题，而是在主持一场车内多人游戏。
- AI 能融合座舱上下文，包括游戏状态、乘客状态、车外环境和车辆状态。
- 模拟乘客能半自主参与，让体验更像真实车内互动。
- 多模态感知信息通过时间轴自动发生，形成更自然的车程变化。
- 急刹、主驾疲惫、高速行驶等高优先级事件可以打断游戏。
- AI 的自然话术和网页 UI 指令通过结构化 JSON 分离，便于后续扩展。

## 统一体验形态

产品不区分“演示模式”和“体验模式”，避免增加交互复杂度。

统一形态是：

```text
可分享体验版
```

它既能用于比赛现场演示，也能分享给部门同事体验。

页面仍保留上帝视角控制能力，但核心体验应围绕“用户参与 + 模拟乘客参与 + AI 主持控场 + 时间轴事件变化”展开。

## 目标用户

主要用户包括：

- AI 应用比赛评委
- 内部同事
- 方案评审人员
- 后续参与共创的产品、设计、算法或工程同事

用户打开页面后，应该能在较短时间内理解：

- 这是一个智能座舱 AI 游戏
- 自己可以作为车内乘客参与
- 其他乘客也会被系统模拟出来
- AI 会根据座舱状态改变游戏节奏

## 核心体验叙事

推荐叙事：

1. 用户打开网页，看到车内四个座位和 AI 猜谜游戏。
2. 用户点击 `开始模拟`，AI 主持人面向车内乘客开场并出题。
3. 用户可以输入提问或答案。
4. 系统会让其他模拟乘客半自主提问或猜答案。
5. 时间轴自动触发车速、环境、乘客状态或车辆事件变化。
6. 当发生急刹等高优先级事件时，游戏立即暂停，AI 转为安全提醒。
7. 状态恢复后，AI 继续控场，并在猜中时揭晓谜底与给出情绪价值。

## 黄金体验线

后续版本应设计几条固定的黄金体验线，确保比赛演示和部门分享时能稳定呈现“AI 有眼力见”的完整故事，而不是只展示零散按钮效果。

黄金体验线建议至少覆盖：

- 高速亲子出行：开场、后排小朋友参与、主驾疲惫、急刹打断、自动恢复、真实副驾或后排小朋友答对。
- 雨天朋友聚会：雨天环境变化、朋友玩梗、模拟乘客接梗、玩家接近答案、AI 给提示、猜对收尾。
- 风景区家庭出游：雪景/风景区入题、长辈稳健提问、有人睡着、轻声继续、快到目的地绝杀局。

每条黄金体验线都应明确默认环境、乘客关系、谜底主题、关键时间轴事件、预期 AI 主持策略、预期 UI 高光表现和备用手动触发方式。

## 首屏轻提示

分享体验版首屏需要给用户一个极轻量的默认提示，降低第一次打开链接的理解成本。

建议提示文案：

```text
点击开始模拟，作为副驾参与一局 15 问猜谜。其他乘客会自动加入。
```

该提示应尽量轻，不做大段教程，不遮挡主体验。

## 产品边界

V1.x 阶段重点做“可体验、可演示、可分享、可扩展”的 demo，不追求量产级车机系统。

当前要做：

- 横屏网页体验
- 以 `1920 x 1080` 为基准的固定舞台布局，整页按浏览器宽度等比缩放
- 三分区布局，左侧为中控屏与手动控制区，右侧为座舱与环境模拟区
- 左侧中控屏内容区保持 16:9，标题栏不计入 16:9 高度
- 右侧座舱、环境背景与融合决策可视化
- 左下手动控制区承载场景切换、开始/暂停/重置模拟、车速、环境、事件和副驾输入
- Coze/Dify Workflow 主链路
- 半自主模拟乘客
- 时间轴模拟多模态事件
- 高优先级事件打断机制
- 结构化输入输出协议
- 后续插件化扩展预留

当前不做：

- 多个独立外部 Agent 并行
- 真正多人在线
- 真实车机系统集成
- 真实摄像头/麦克风/车辆传感器接入
- 真实语音输入和 TTS
- 完整账号体系
- 真实长期记忆
- 完整排行榜
- 大规模题库运营后台
- 手机端适配

## 设计原则

### Workflow 主路径

正式体验以 Workflow 为主路径。Mock 只作为开发期和异常保护，不作为对外体验重点。

### 半自主而非完全自由

模拟乘客不采用死剧本，也不采用多个完全自由 Agent。第一阶段采用“人设 + 发言意图 + 状态判断 + 随机选择 + Workflow 改写”的半自主机制。

### 时间轴驱动真实感

车速、环境、乘客情绪、车辆事件等多模态感知信息通过预设时间轴自动发生。手动按钮仍保留，用于临场控制和调试。

目的地和乘客关系优先由黄金体验线和时间轴决定，不在手动控制区提供手动切换。这样体验者更像是在进入一条完整车程，而不是调试一堆散点参数。

### 输入始终可用

车内互动输入栏固定在手动控制区底部。真实用户座位固定为 `副驾`，不提供座位下拉选择，避免体验者误以为需要扮演多名乘客。其他座位由时间轴或 Workflow 触发的模拟乘客参与。

当前端已有请求在处理中，新输入进入队列，等待上一轮结束后继续提交。暂停模拟时，除 `继续模拟` 和 `重置模拟` 外其他手动控制不可用；急刹事件由前端自动恢复，不再提供手动 `恢复游戏` 按钮。

座位本身不承载“当前选择态”。只有答对时，对应乘客座位旁的氛围灯亮起，作为明确的胜利反馈。

### 结构化输出

AI 说的话、模拟乘客动作和 UI 动作必须分开。网页不能靠解析自然语言决定界面变化。

### 融合决策可见

Demo 需要让观众看见 AI 不是在随机聊天，而是在执行“感知 -> 决策 -> 执行”链路。

可视化表达应轻量，例如：

```text
感知：主驾疲惫 / 高速行驶 / 后排睡着
决策：降低主驾参与 / 轻声继续 / 暂停游戏
执行：AI 话术 + 座舱模式 + UI 高光
```

这部分可以先作为座舱与环境模拟区的一块轻量信息，不做复杂日志系统。

V1.2 起，融合决策可视化主面板放在座舱与环境模拟区，让观众在同一个视觉区域里看到“发生了什么”和“AI 为什么这样做”。手动控制区只保留体验线、车速、车外环境和事件控制。

### 稳定可分享

对外分享版本必须尽量减少用户理解成本，避免暴露敏感密钥、复杂调试字段或过多上帝视角操作。

分享体验中的默认入口应轻量直接：用户看到首屏提示后点击 `开始模拟`，作为副驾参与，其他乘客自动加入。AI 主持人的气泡只面向车内乘客，不出现“点击按钮”“时间轴”等出戏表述。

### 插件化预留

猜谜是第一版游戏插件。后续游戏应复用座舱状态、时间轴事件、模拟乘客机制和 Workflow 契约。

## V1.0 已完成基线

V1.0 已固定为 Git 标签：

```text
V1.0
```

V1.0 包含：

- 静态网页骨架
- 三分区布局
- 2D 座舱占位
- AI 主持人占位
- 10 条本地题库
- 玩家提问
- 急刹打断
- 急刹后自动恢复
- 猜对揭晓
- Workflow 接口输入框
- 本地兜底决策

V1.0 是代码和交互骨架基线。V1.1 之后的需求方向升级为可分享体验版。

## 成功标准

短期成功标准：

- 用户打开链接后 30 秒内知道可以参与猜谜。
- 用户固定作为副驾参与时，其他模拟乘客也能自然参与。
- AI 主持人能基于座舱上下文控场。
- 时间轴事件能自动改变车速、环境、乘客状态或车辆状态。
- 急刹打断和自动恢复稳定可用。
- Workflow 返回的 `ai_reply_text`、`passenger_action` 和 `ui_change` 能驱动页面表现。

中期成功标准：

- 可部署为公网链接并分享给部门同事体验。
- 至少支持 3 条预设时间轴。
- 至少支持 3 种乘客关系下的模拟乘客风格。
- 支持至少 5 类座舱感知事件。
- 保持后续扩展到其他游戏插件的架构空间。
