# V1.4 UI 设计输入包

## 文档目的

这份文档服务 `V1.4：整体 UI 与交互体验优化`。

它不是完整静态原型，也不是视觉规范终稿，而是一份可直接用于以下工作的统一输入：

- 给设计师或自己整理视觉方向
- 给 ChatGPT 生成界面效果图 Prompt
- 给 Codex 落地网页 UI
- 给后续资源制作确定优先级

目标是让当前 demo 从“功能闭环稳定”升级到“比赛现场第一眼可信、分享出去也愿意多玩几分钟”。

## V1.4 的版本目标

V1.4 不新增大功能，重点做三件事：

1. 让界面看起来更像一个可信的智能座舱体验，而不是调试原型
2. 让游戏信息、乘客状态、AI 主持人、融合决策之间的关系更清楚
3. 让关键交互更顺，包括等待、打断、答对、暂停、恢复、思考等状态

一句话概括：

```text
在不大改产品结构的前提下，把现有 demo 做到“比赛可打、分享可看、体验不出戏”。
```

## 不在 V1.4 解决的内容

以下内容放到后续版本，不在 V1.4 里硬塞：

- Coze 主持话术深度优化
- Know-how 体系继续扩充
- 多游戏插件扩展
- 真正量产级车机视觉系统
- 真实 TTS / ASR
- 手机端适配
- 完整乘客精细插画体系

也就是说，V1.4 解决的是“观感和交互”，不是“AI 智能本身”。

## 设计总原则

### 1. 车机感优先，演示感次之

界面气质应更接近智能座舱中控，而不是营销页、游戏海报或概念大屏。

关键词：

- 克制
- 清晰
- 高级但不过分炫技
- 信息明确
- 有轻微未来感
- 带一点 demo 演示高光，但不浮夸

### 2. 三个区域要有主次，不要平均发力

当前页面分为：

- 中控屏
- 座舱与环境模拟区
- 手动控制区

主次建议：

- 第一视觉焦点：中控屏
- 第二视觉焦点：座舱与环境模拟区
- 第三视觉焦点：手动控制区

手动控制区不应该抢戏，它更像一个演示辅助台。

### 3. “融合决策”要可见，但不应像日志后台

观众需要知道 AI 不是随机说话。

但这块信息应该像“轻量决策卡片”，而不是调试输出窗口。要让人一眼看懂：

```text
发生了什么 -> AI 为什么这样做 -> 页面做了什么
```

### 4. 状态比装饰更重要

V1.4 的重点不是大面积装饰，而是让这些状态变得更明确：

- AI 思考中
- 玩家提问中
- 模拟乘客参与中
- 安全暂停
- 轻声继续
- 接近答案
- 答对收尾

### 5. 尽量少切图，优先代码化

V1.4 不建议先出一套重切片设计稿再开发。

推荐策略：

- 布局、卡片、按钮、芯片、气泡、光效、基础动画：代码实现
- 环境背景：图片资源
- AI 主持人形象：1 张占位图或 1 组轻状态图
- 乘客头像状态：先代码占位，后续可替换为切图

## 目标用户的第一印象

用户打开页面后的前 5 秒，应感受到：

1. 这是一个车内 AI 游戏，不是普通网页问答
2. 这里真的有“AI 主持人 + 车内乘客 + 车外环境”
3. 我可以直接点击开始参与
4. 系统会自己演起来，不需要我理解很多按钮

## 视觉关键词

建议使用以下关键词约束视觉探索：

- 智能座舱
- 中控屏
- 轻科技
- 冷静、清透
- 白天车内自然光
- 低饱和背景 + 少量绿色高亮
- 信息层级清楚
- 微光感
- 安全感

避免这些方向：

- 纯营销大屏风
- 紫蓝渐变科技海报风
- 过重拟物
- 过多装饰线条
- 过强游戏化皮肤
- 太像开发后台

## 核心页面定义

V1.4 先围绕 3 个核心画面做设计和实现，不追求一次把所有边角状态都画全。

### 画面 A：待开始

用途：

- 用户第一次打开链接时看到的默认状态

需要传达的信息：

- 当前是一个智能座舱猜谜体验
- 用户将以副驾身份参与
- 其他乘客会自动加入
- 点击开始即可进入

页面要点：

- 中控屏中央显示“游戏待开始”
- AI 主持人处于待命态，但不说出戏的话
- 座舱区乘客处于安静默认状态
- 手动控制区只显示：
  - 当前场景名
  - 场景切换
  - 开始模拟
  - 轻提示文案

重点优化：

- 首屏不要空
- 不要像系统待开发
- 开始按钮要足够明确，但不粗暴

### 画面 B：游戏进行中

用途：

- 最常见、停留时间最长的主体验画面

需要传达的信息：

- 当前游戏正在进行
- AI 正在主持
- 用户和其他乘客都在参与
- 谜底还没揭晓，但有提示和进度

页面要点：

- 中控屏中央展示：
  - 当前阶段文案，如“提示”
  - 谜题提示文本
  - 回合进度和提问计数
- AI 主持人气泡应是主阅读区之一
- 座舱区乘客有基础表情和少量活动状态
- 决策卡片显示最近一次“感知 -> 决策 -> 执行”
- 手动控制区显示：
  - 车速
  - 环境切换
  - 事件按钮
  - 暂停 / 重置
  - 副驾输入框

重点优化：

- 视觉焦点要稳定，不要每块都像主角
- 用户输入区要明显可用
- 乘客状态有变化，但不能太吵
- 等待 Workflow 返回时，AI 思考态要自然

### 画面 C：特殊事件 / 答对收尾

用途：

- 表现 demo 的高光时刻

这一类画面包括两种子状态：

#### C1. 安全/状态打断

典型场景：

- 急刹
- 主驾疲惫
- 有人睡着

需要传达的信息：

- 游戏不是一直聊，它会被座舱状态打断
- AI 会优先处理安全和情境

表现重点：

- 中控屏节奏短暂停顿
- 气氛转为更克制、更明确
- 座舱状态有明显变化
- 决策卡片更容易看懂

#### C2. 答对收尾

典型场景：

- 用户答对
- 模拟乘客答对

需要传达的信息：

- 谁答对了
- AI 主持人给到情绪价值
- 谜底正式揭晓

表现重点：

- 只有答对乘客座位亮起氛围灯
- 有轻量胜利音效
- AI 说出简短但有情绪价值的总结
- 座舱、主持人、中控屏三处反馈要一致

## 状态视觉优先级

不是所有状态都要做成独立插画。V1.4 先按优先级处理。

### P0：必须清楚

- AI 思考中
- 安全暂停
- 答对
- 睡着
- 主驾疲惫

### P1：应该清楚

- 大笑
- 沉默
- 后排参与提问
- 接近答案
- 快到目的地

### P2：可先占位

- 轻声继续
- 主持人不同情绪的细微变化
- 更细分的乘客人设差异

## 资源清单

V1.4 推荐把资源拆成三类。

### A 类：必须出图资源

这些适合先做静态效果图或图片资源：

- 环境背景图 3 到 5 张
  - 高速晴天
  - 城区雨天
  - 风景区雪景
  - 可选：高速深夜
  - 可选：风景区晴天
- AI 主持人占位形象 1 套
- 可选的乘客头像风格参考 1 套

### B 类：代码直接实现

- 按钮
- 输入框
- 芯片
- 卡片
- 决策面板
- 气泡
- 高亮边框
- 思考态
- 暂停态遮罩
- 胜利光效

### C 类：后补资源

- 乘客不同情绪切图
- 更精细的主持人状态图
- 答对庆祝小动效素材
- 音效资源替换版

## 没有完整静态原型图时的工作流

当前不需要先补一整套高保真静态原型。推荐流程如下：

### 第 1 步：先出 2 到 3 个整体视觉方向

建议只探索这些差异：

- 更车机克制
- 更演示高光
- 更家庭陪伴感

每个方向只需要覆盖同一张主画面，不要一开始画全流程。

### 第 2 步：选定一个方向后，补 3 个关键状态

- 待开始
- 游戏进行中
- 事件/答对高光

### 第 3 步：确定哪些是图片，哪些直接代码

不要反过来。

先确认：

- 背景图是否需要真实图片
- 主持人是否需要插画
- 乘客是否需要切图

其余优先用代码完成。

### 第 4 步：在真实页面里二次调整

即使有了效果图，也不要要求 1:1 还原到像设计稿。

这个项目更适合：

```text
效果图定方向 -> 代码里调布局/状态/动效 -> 再截图回看
```

## ChatGPT 与 Codex 的分工建议

### 更适合交给 ChatGPT 的事情

- 生成视觉方向描述
- 写效果图 Prompt
- 探索 2 到 3 个界面概念方向
- 给出风格板关键词
- 辅助整理状态文案和界面气质

### 更适合交给 Codex 的事情

- 把选中的方案真正实现到网页
- 调整三分区比例与固定舞台布局
- 实现思考态、暂停态、答对态、气泡节奏
- 做乘客状态占位样式
- 做视觉回归和交互修补

结论：

```text
ChatGPT 负责“先想像”，Codex 负责“把它做出来并调通”。
```

## 效果图生成建议

如果要用 AI 先出图，建议每次只出一张主画面，不要一次让它画整个流程。

优先出以下三张：

1. 待开始页
2. 游戏进行中页
3. 答对高光页

每张图都要明确这些信息：

- 画面用途
- 三分区结构
- 主视觉焦点
- 关键词
- 哪些内容是功能信息，哪些只是气质参考

## 可直接给 ChatGPT 的出图 Prompt 模板

```text
请为一个“智能座舱 AI 猜谜 Demo”生成网页效果图描述，用于 16:9 横屏中控体验。

产品背景：
- 这是一个可分享体验的智能座舱网页 demo
- 用户以副驾身份参与 15 问猜谜
- AI 主持人主持游戏
- 其他乘客会自动参与
- 页面包含三个区域：中控屏主体验区、座舱与环境模拟区、手动控制区

这次请只画一个状态：{待开始 / 游戏进行中 / 答对高光}

视觉要求：
- 气质接近智能座舱中控，不要像营销页
- 清透、克制、轻科技
- 低饱和背景，少量绿色高亮
- 信息层级清楚
- 可比赛展示，但不过度炫技

结构要求：
- 中控屏为第一视觉焦点
- 座舱区能看到 4 个座位、乘客、环境背景
- 手动控制区信息密度较低，不抢戏
- 需要体现 AI 主持人、乘客状态、游戏提示、决策可视化

请输出：
- 1 个完整视觉方向
- 关键视觉说明
- 布局说明
- 组件说明
- 可直接用于进一步实现的设计描述
```

## V1.4 交付物建议

V1.4 不要求先有完整 Figma，再开始开发。

建议交付物是：

1. 一份选定的视觉方向说明
2. 三张关键状态效果图
3. 一份资源清单
4. 一轮真实网页落地
5. 一轮视觉回归修订

## 开工前确认项

进入 V1.4 实现前，建议只确认这 5 件事：

1. 最终选哪一个整体视觉方向
2. AI 主持人是否先继续用占位形象
3. 环境背景图先用真实图片还是 AI 生成图
4. 乘客状态先用代码表情还是先做切图
5. 融合决策卡片是更偏“说明卡”还是更偏“状态卡”
