# 版本路线图

## 总体策略

后续开发按小版本推进，每个版本完成后都进行 Git 提交，并打版本标签。

V1.0 是应用骨架基线。V1.1 之后的目标升级为“可分享体验版”，不再区分演示模式和体验模式。

## 当前判断

截至 2026-06-24，项目的真实进度判断如下：

- `V1.0`：完成
- `V1.1`：完成
- `V1.2`：核心体验闭环已完成，三条黄金体验线、固定舞台、时间轴、模拟乘客和融合决策可视化均已落地
- `V1.3`：进行中，当前重点已经从“能不能跑起来”转为“公网一致性、话术智能感、比赛演示稳定性”

也就是说，接下来最重要的不是再塞新功能，而是把 `V1.3` 做扎实。

## 接下来 3 步

### 第 1 步：减少前端主持话术覆盖，让 Coze 更像真正主持人

目标：

- 降低重复度
- 让正常主持话术更多来自 Coze，而不是前端写死

工作项：

- 依据 `HOST_COPY_OWNERSHIP.md` 梳理前端覆盖边界
- 保留安全接管、真实副驾主流程保护、超时 fallback
- 优先把 `makeVictoryHostText()`、普通问答 fallback、环境融合表达逐步放权给 Coze
- 给 Coze 增加高频策略的话术轮换要求：`S00`、`S02`、`S14`、`S21`

产出：

- 前端主持话术覆盖收缩一版
- Coze 提示词/策略文档更新一版

### 第 2 步：解决公网版本不一致，形成稳定可分享链接

目标：

- 本地最新版和公网体验一致
- 部门同事点开链接时就是当前稳定版本

工作项：

- 重新打通 Vercel 生产部署链路
- 确认 `/api/workflow` 代理、环境变量和鉴权方式稳定
- 建立“本地锚点 -> 公网验收”最短发布路径
- 每次稳定版本后同步更新 `PUBLIC_ACCEPTANCE_REPORT.md`

风险：

- 当前环境到 Vercel 的 TLS/网络握手不稳定
- 公网部署一旦滞后，演示反馈会和本地版本错位

### 第 3 步：比赛导向收口

目标：

- 演示 3 到 5 分钟内稳定、顺滑、像真人主持

工作项：

- 固定 1 条主演示线，另外 2 条作为备用线
- 把比赛最常见的问题补进 `TEST_SCRIPT.md`
- 收紧气泡节奏、庆祝时机、乘客表情和安全打断表现
- 梳理一份“比赛现场操作顺序 + 兜底说法”

产出：

- 一版比赛验收脚本
- 一版分享体验脚本
- 一版演示现场风险清单

建议标签：

- `V1.0`：第一版骨架，已完成
- `V1.1`：Workflow + 模拟乘客最小版
- `V1.2`：体验完善与时间轴增强
- `V1.3`：分享部署与视觉优化
- `V1.4`：Coze 主持智能化与比赛版收口
- `V1.5`：插件化预备与第二游戏最小版
- `V2.0`：多游戏插件版本

## V1.0：应用骨架

状态：已完成。

目标：

- 搭建三分区网页
- 跑通最小闭环
- 留出 Workflow 接口
- 保留基础 Mock 能力

已包含：

- 静态网页
- 2D 座舱占位
- AI 主持人占位
- 10 条本地题库
- 玩家提问
- 急刹打断与恢复游戏
- 猜对揭晓
- Workflow URL 输入
- 本地兜底决策

## V1.1：Workflow + 模拟乘客最小版

状态：已完成，公网稳定性验收通过。锚点建议使用 `V1.1-accepted`。

目标：

尽快让体验从“用户单人手动模拟”变成“用户 + AI 主持人 + 至少一位模拟乘客”的真实车内互动。

工作项：

- 接入真实 Coze/Dify Workflow
- 对齐新版 Workflow 契约
- 支持 `passenger_action`
- 支持 `trigger_type = simulation`
- 新增车内互动输入区
- 用户输入固定为副驾真实用户
- 模拟乘客发言后，在对应座位显示气泡
- 互动区追加用户、模拟乘客、AI 主持人消息
- 增加至少 1 条基础时间轴
- 时间轴可触发车速、环境、乘客状态或事件变化
- 保留手动控制区用于多模态状态和事件调试

验收标准：

- 点击开始模拟会请求 Workflow 的 `start_game` 事件
- 用户提问会请求 Workflow
- Workflow 可返回 `passenger_action`
- 页面能展示模拟乘客发言
- AI 主持人能回应用户或模拟乘客
- 急刹事件仍能立即暂停游戏
- 游戏暂停时不生成模拟乘客主动发言
- 至少一条时间轴可自动触发事件

风险：

- Workflow 返回字段不稳定
- 模拟乘客过早猜中或重复提问
- 时间轴事件与用户手动操作冲突
- 接口跨域或鉴权问题

建议：

- 第一阶段只支持单个 `passenger_action`
- 不做多个独立外部 Agent
- 模拟乘客动作由统一 Workflow 决定
- Workflow 输出必须加 JSON 校验节点

## V1.2：体验完善与时间轴增强

状态：已基本完成。当前仅保留少量体验细节修补，不再作为主要开发主线。

目标：

让部门同事打开链接后，不需要演示者辅助，也能自然体验一局游戏。

工作项：

- 设计 3 条固定黄金体验线
- 实现 3 条黄金体验线切换：高速亲子出行、雨天朋友聚会、风景区雪景家庭
- 页面改为 `1920 x 1080` 固定舞台，整页按宽度优先等比缩放
- 将布局调整为左侧中控屏+手动控制区、右侧座舱与环境模拟区
- 中控屏内容区保持 `16:9`，标题栏不计入 16:9 高度
- 将融合决策可视化面板放入右侧座舱模拟区
- 将副驾输入栏固定在左下手动控制区底部
- 手动控制区移除目的地、乘客关系和 Workflow 代理可视配置
- 目的地和乘客关系由当前黄金体验线/时间轴决定
- 真实用户固定为副驾，不提供座位下拉选择
- 座舱座位不展示选择状态
- Workflow 忙时用户输入进入前端队列，避免抢话或丢话
- 增加 3 条以上预设时间轴
- 时间轴支持开始、暂停/继续、重置、黄金体验线切换
- 增加模拟乘客人设配置
- 增加不同乘客关系下的模拟乘客风格
- 增加后排小朋友更跳脱、更好奇的模拟乘客风格
- 增加防重复提问逻辑
- 增加防过早猜中逻辑
- 增加暂停时阻止模拟乘客发言逻辑
- 确保睡着乘客不会继续主动发言
- 增加主驾高速降频逻辑
- 增加轻量首屏提示
- 增加融合决策可视化面板第一版
- 优化手动控制区的信息密度
- 急刹事件前端自动恢复，不再提供手动恢复按钮
- 答对乘客座位氛围灯亮起
- 增加轻量答对三音提示
- 弱化 Mock 对外存在感

验收标准：

- 至少 3 条黄金体验线可以切换，其中至少 1 条可稳定自动跑通
- 页面在 16:9 横屏窗口内按固定舞台稳定展示，无局部错位
- 用户打开页面后默认作为副驾参与，无需选择座位
- 首屏能看到轻量提示，知道点击开始模拟即可参与
- 其他座位能由模拟乘客参与
- 时间轴能自动制造车程变化
- 模拟乘客不会频繁重复或抢话
- 睡着乘客不会继续主动发言
- 融合决策面板能展示最近一次感知、决策和执行
- 用户输入在模拟乘客发言或 Workflow 响应期间不被禁用
- 答对时对应座位有明确氛围灯高光
- 用户能完成一局自然猜谜
- Workflow 异常时页面不会崩溃

当前已完成：

- 三条黄金体验线切换已完成
- 固定舞台、左右布局和中控屏 `16:9` 已完成
- 副驾固定输入、时间轴开始/暂停/重置已完成
- 模拟乘客基础参与已完成
- 急刹、睡着、主驾疲惫、快到目的地等核心事件已完成
- 答对座位高亮、三音提示、思考态 UI、乘客基础表情已完成
- 用户输入排队与最短气泡展示保护已完成

剩余建议仅作为体验优化项：

- 继续细化乘客贴图和表情资源
- 继续减少重复主持话术
- 继续优化个别时间轴节点衔接

- 优化乘客贴图

新增当前主线工作项：

- 清理前端对 Coze 主持话术的非必要覆盖
- 提升 Coze 正常问答、多乘客 cue、答对总结的表达多样性

## V1.5：插件化预备与第二游戏最小版

状态：待开始。建议在 `V1.4` 稳定后推进。

目标：

把“猜谜”从单游戏 demo 升级为“可扩展的智能座舱游戏底座”，但先不追求完整平台化。

工作项：

- 将猜谜逻辑整理为 `RiddlePlugin`
- 将题库独立为数据文件
- 将 Workflow 输入构建拆为通用部分和插件部分
- 将 Workflow 输出应用拆为通用部分和插件部分
- 将模拟乘客约束整理为通用能力
- 增加插件注册表
- 新增第 2 个最小游戏插件
  建议优先：`观察挑战` 或 `成语接龙`

建议接口：

```ts
GamePlugin = {
  id,
  name,
  getInitialState,
  buildWorkflowGamePayload,
  applyWorkflowOutput,

  getSimulationContext
}
```

验收标准：

- 猜谜仍能完整跑通
- 主应用不直接依赖猜谜细节
- 第二个游戏插件可在同一座舱框架下跑通最小闭环

## 部署路线

### 阶段1：静态部署

适合 V1.0-V1.2 早期验证。

可选：

- Vercel
- Netlify
- GitHub Pages
- 阿里云 OSS 静态网站
- 公司内部静态资源服务器

限制：

- 前端直接请求 Workflow，可能有跨域和密钥问题

### 阶段2：轻量后端代理

适合 V1.3 分享版本。

用途：

- 隐藏 Coze/Dify 密钥
- 解决跨域
- 做接口日志
- 做返回格式校验
- 做超时和异常提示

### 阶段3：正式分享版本

要求：

- 固定公网 URL
- 固定 Workflow 版本
- 默认场景可直接体验
- 接口异常有清晰提示
- 准备比赛现场备用版本

## Git 版本锚点规则

每个稳定版本建议：

1. 完成功能
2. 跑测试脚本
3. 更新文档
4. Git 提交
5. 打版本标签

示例：

```bash
git add .
git commit -m "V1.1 workflow and passenger simulation"
git tag -a V1.1 -m "V1.1 workflow and passenger simulation"
```

## 比赛前冻结策略

建议比赛前至少 1 天冻结功能。

冻结后只允许：

- 修明显 bug
- 调整文案
- 替换图片
- 修改时间轴配置
- 修改演示讲法

不建议临近比赛再做：

- 大重构
- 换框架
- 改接口契约
- 新增多个外部 Agent
- 新增未经测试的外部依赖
