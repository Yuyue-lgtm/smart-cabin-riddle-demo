# 版本路线图

## 总体策略

后续开发按小版本推进，每个版本完成后都进行 Git 提交，并打版本标签。

V1.0 是应用骨架基线。V1.1 之后的目标升级为“可分享体验版”，不再区分演示模式和体验模式。

建议标签：

- `V1.0`：第一版骨架，已完成
- `V1.1`：Workflow + 模拟乘客最小版
- `V1.2`：体验完善与时间轴增强
- `V1.3`：分享部署与视觉优化
- `V1.4`：插件化结构整理
- `V2.0`：多游戏插件版本

## V1.0：应用骨架

状态：已完成。

目标：

- 搭建三分区网页
- 跑通最小闭环
- 留出 Workflow 接口
- 保留基础 Mock 能力

已包含：

- 静态网页
- 2D 座舱占位
- AI 主持人占位
- 10 条本地题库
- 玩家提问
- 急刹打断
- 恢复游戏
- 猜对揭晓
- Workflow URL 输入
- 本地兜底决策

## V1.1：Workflow + 模拟乘客最小版

状态：已完成，公网稳定性验收通过。锚点建议使用 `V1.1-accepted`。

目标：

尽快让体验从“用户单人手动模拟”变成“用户 + AI 主持人 + 至少一位模拟乘客”的真实车内互动。

工作项：

- 接入真实 Coze/Dify Workflow
- 对齐新版 Workflow 契约
- 支持 `passenger_action`
- 支持 `trigger_type = simulation`
- 左侧区域底部新增车内互动区
- 将用户输入从右下控制区迁移到左侧底部
- 模拟乘客发言后，在对应座位显示气泡
- 互动区追加用户、模拟乘客、AI 主持人消息
- 增加至少 1 条基础时间轴
- 时间轴可触发车速、环境、乘客状态或事件变化
- 保留右下控制区用于多模态状态和事件调试

验收标准：

- 点击 AI 开场会请求 Workflow
- 用户提问会请求 Workflow
- Workflow 可返回 `passenger_action`
- 页面能展示模拟乘客发言
- AI 主持人能回应用户或模拟乘客
- 急刹事件仍能立即暂停游戏
- 游戏暂停时不生成模拟乘客主动发言
- 至少一条时间轴可自动触发事件

风险：

- Workflow 返回字段不稳定
- 模拟乘客过早猜中或重复提问
- 时间轴事件与用户手动操作冲突
- 接口跨域或鉴权问题

建议：

- 第一阶段只支持单个 `passenger_action`
- 不做多个独立外部 Agent
- 模拟乘客动作由统一 Workflow 决定
- Workflow 输出必须加 JSON 校验节点

## V1.2：体验完善与时间轴增强

目标：

让部门同事打开链接后，不需要演示者辅助，也能自然体验一局游戏。

工作项：

- 设计 3 条固定黄金体验线
- 增加用户座位选择
- 增加 3 条以上预设时间轴
- 时间轴支持开始、暂停、重置、切换
- 增加当前时间轴事件提示
- 增加模拟乘客人设配置
- 增加不同乘客关系下的模拟乘客风格
- 增加后排小朋友更跳脱、更好奇的模拟乘客风格
- 增加防重复提问逻辑
- 增加防过早猜中逻辑
- 增加暂停时阻止模拟乘客发言逻辑
- 确保睡着乘客不会继续主动发言
- 增加主驾高速降频逻辑
- 增加轻量首屏提示
- 增加融合决策可视化面板第一版
- 优化左侧车内互动区的信息密度
- 弱化 Mock 对外存在感

验收标准：

- 至少 1 条黄金体验线可以稳定自动跑通
- 用户打开页面后可以选择或默认进入一个座位
- 首屏能看到轻量提示，知道点击 AI 开场即可参与
- 其他座位能由模拟乘客参与
- 时间轴能自动制造车程变化
- 模拟乘客不会频繁重复或抢话
- 睡着乘客不会继续主动发言
- 融合决策面板能展示最近一次感知、决策和执行
- 用户能完成一局自然猜谜
- Workflow 异常时页面不会崩溃

## V1.3：分享部署与视觉优化

目标：

形成可发给部门体验的公网版本，并提升第一眼观感。

工作项：

- 部署公网链接
- 处理 Workflow 鉴权与跨域
- 必要时增加轻量后端代理
- 隐藏或保护敏感接口信息
- 固定默认体验场景
- 优化座舱插画
- 替换环境占位背景
- 优化乘客贴图
- 优化 AI 主持人形象
- 增加胜利、安全暂停、环境切换的视觉反馈
- 增加答对乘客座位氛围灯
- 增加答对音效
- 增加猜对后的 AI 高情绪价值总结视觉呈现
- 优化横屏大屏布局细节

验收标准：

- 部门同事可通过链接直接访问
- 不暴露长期密钥
- 默认场景可直接开始体验
- 第一眼能看出车内座舱和多人互动
- 答对时有明确座位高光、音效和 AI 总结
- 视觉升级不破坏核心流程

## V1.4：插件化整理

目标：

把猜谜从主应用逻辑里拆出来，为后续游戏扩展做准备。

工作项：

- 抽象 `GamePlugin` 接口
- 将猜谜逻辑整理为 `RiddlePlugin`
- 将题库独立为数据文件
- 将 Workflow 输入构建分为通用部分和插件部分
- 将 Workflow 输出应用分为通用部分和插件部分
- 将模拟乘客约束整理为通用能力
- 增加插件注册表

建议接口：

```ts
GamePlugin = {
  id,
  name,
  getInitialState,
  buildWorkflowGamePayload,
  applyWorkflowOutput,
  renderGameState,
  getSimulationContext
}
```

验收标准：

- 猜谜仍能完整跑通
- 主应用不直接依赖猜谜细节
- 新插件可以复用座舱状态、时间轴和模拟乘客机制

## V2.0：多游戏插件

目标：

从“猜谜 demo”升级为“智能座舱 AI 游戏平台 demo”。

候选插件：

- 成语接龙
- 猜歌
- 故事接龙
- 观察挑战
- 找找看

工作项：

- 增加插件切换
- 每个插件有独立玩法状态
- Workflow 根据 `plugin_id` 分流
- 座舱事件对不同插件有不同影响
- 模拟乘客能适配不同游戏玩法

验收标准：

- 至少两个游戏插件可切换
- 切换插件不破坏座舱状态
- 高优先级车辆事件对所有插件生效
- 模拟乘客机制可复用

## 部署路线

### 阶段1：静态部署

适合 V1.0-V1.2 早期验证。

可选：

- Vercel
- Netlify
- GitHub Pages
- 阿里云 OSS 静态网站
- 公司内部静态资源服务器

限制：

- 前端直接请求 Workflow，可能有跨域和密钥问题

### 阶段2：轻量后端代理

适合 V1.3 分享版本。

用途：

- 隐藏 Coze/Dify 密钥
- 解决跨域
- 做接口日志
- 做返回格式校验
- 做超时和异常提示

### 阶段3：正式分享版本

要求：

- 固定公网 URL
- 固定 Workflow 版本
- 默认场景可直接体验
- 接口异常有清晰提示
- 准备比赛现场备用版本

## Git 版本锚点规则

每个稳定版本建议：

1. 完成功能
2. 跑测试脚本
3. 更新文档
4. Git 提交
5. 打版本标签

示例：

```bash
git add .
git commit -m "V1.1 workflow and passenger simulation"
git tag -a V1.1 -m "V1.1 workflow and passenger simulation"
```

## 比赛前冻结策略

建议比赛前至少 1 天冻结功能。

冻结后只允许：

- 修明显 bug
- 调整文案
- 替换图片
- 修改时间轴配置
- 修改演示讲法

不建议临近比赛再做：

- 大重构
- 换框架
- 改接口契约
- 新增多个外部 Agent
- 新增未经测试的外部依赖
